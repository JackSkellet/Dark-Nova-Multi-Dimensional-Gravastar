## Tested Methods

- **E1 contextual routing:** synthetic polysemous rows are routed by static token rules, flat contextual keyword rules, hierarchical broad-then-specific rules, and random control. Quality is route-label accuracy; cost is active component count, estimated traffic, expert balance, and router latency.
- **E1b robust routing:** synthetic single-route, mixed-route, and adversarial-cue rows are routed by single-label contextual, explicit multi-label, and latent-centroid multi-label routers. Quality is exact route-set accuracy, recall, false positives, and adversarial accuracy.
- **E2 storage:** a synthetic 64x64 matrix is encoded by FP32, FP16, uniform int8/int4, low-rank SVD, low-rank plus sparse residual, rank-1 outer product, shared basis, tensorized two-block, product-quantized rows, Kronecker rank-1, and Tensor Train 4D. Total bytes include metadata/codebook/index overhead where implemented.
- **E3-E3k selective precision:** synthetic linear carriers, tiny transformer-style output heads, one random internal MLP matrix, one trained internal MLP matrix, one real open-model checkpoint matrix, one real-checkpoint token-id task-loss smoke test, one real-checkpoint natural-text embedding task-loss smoke test, and one real-checkpoint natural-text internal-tensor task-loss smoke test are quantized with uniform/groupwise low-bit baselines plus random, output-error-selected, BF16-like, and FP32 protected rows.
- **E4-E4d lookup/runtime:** component banks are routed by dense/exact/tree/hash/cache methods, then by fuller CPU routed-execution simulation, then by a Torch batched exact path with authorization masking and selected-component dispatch, then by an isolated ROCm transfer-scaling benchmark.
- **E5-E5g continual/repository tasks:** synthetic API facts, trainable low-rank adapter deltas, generated stale-doc histories, generated patch candidates, public Git histories, public generated patch candidates, and historical patch controls are evaluated chronologically without future-commit leakage.
- **E6a-E6i external memory:** a synthetic repository-convention corpus is answered by frozen parametric, text-retrieval, gated text-retrieval, and gated structured external-memory methods. Local public `kilo` and `markupsafe` clones are then used for symbol-definition file QA, defining signature-line QA, canonical Python call-stub generation, minimal Python function-skeleton generation, first-line documented skeleton generation, Markdown API-reference generation, and exact API-doc coverage QA with frozen parametric, text/name-only/signature-only/source-symbol-only, and structured-memory baselines. A synthetic Git-history positive control then tests API-doc drift detection when a source symbol is renamed but a Sphinx directive still references the old API.
- **S1/S2/S2b/S2c security:** deterministic authorization, synthetic canaries, prompt-injection quarantine, poisoning rejection, tamper detection, audit logs, rollback, simple canary encoding detection, separator-normalized stored-instruction detection, trust-label quarantine, and deterministic semantic exfiltration query denial are tested in synthetic harnesses.

## Results Against Hypotheses

- H1: partially supported in a toy setting. Context beats static token routing; hierarchy reduced active components but increased routing latency. E1b shows single-label routing fails mixed-topic prompts, naive multi-label routing adds false positives under overlapping/adversarial cues, and latent-centroid multi-label routing fixes the toy set at higher latency.
- H2: rejected for the tested matrix. Uniform int8 dominated the compositional storage variants, including product-quantized rows, Kronecker rank-1, and Tensor Train 4D after codebook, factor, assignment, and metadata overhead were counted.
- H3: mixed but improved. The first causal-protection policy failed against random protection, E3b succeeded on a synthetic linear task, E3c showed output-error-selected FP32 output-head rows beating random controls on held-out prompts in a random tiny transformer-style model, and E3d repeated the test with a trained output head. E3e showed BF16-like selected carrier rows nearly matching FP32 selected carrier rows at lower byte cost in the same trained output-head setup. E3f moved selection to an internal MLP matrix and beat the random protected mean on held-out logit MSE at the same byte budget, but NLL remained mixed. E3g trained the internal MLP output matrix itself; selected BF16/FP32 protected rows beat matched random controls and groupwise int4 on logit MSE, but BF16-like rows had slightly better logit MSE than FP32 at lower bytes in this seed. E3h used a pinned public `sshleifer/tiny-gpt2` tensor and selected protected rows beat random controls on reconstruction MSE, but row-wise metadata made groupwise and protected-row layouts larger than FP32 for the skinny `[50257, 2]` matrix. E3i used a local minimal GPT-2 forward pass and held-out next-token KL on deterministic token-id sequences; selected protected rows did not beat matched random protected means. E3j replaced those token IDs with byte-level GPT-2 BPE tokenized natural-language repository/code prose; selected protected rows beat the random protected means on held-out KL, but not the best random draws, and the protected layouts still lost on storage. E3k moved the natural-text run to `transformer.h.0.mlp.c_fc.weight` with all-row candidates; uniform/groupwise quantization became smaller than FP32, but all policies had zero measurable KL difference, so selective precision had no task-loss signal.
- H4: mixed. The initial isolated lookup rejected approximate/hash indexing at tested bank sizes. E4b shows exact indexing and cache reuse can beat dense routing in a larger-bank full-path CPU simulation, while hash indexing remains unattractive due to recall and latency. E4c requests ROCm through PyTorch HIP and exercises a real ROCm-backed Torch dispatch path. E4d separately records ROCm transfer scaling across 1 MiB, 8 MiB, and 32 MiB payloads. These prove the ROCm path runs on the local RX 9070 XT and provide transfer baselines, but they are still too small and incomplete to support a speed or architecture claim.
- H5: partially supported as a governance/control simulation and public-repo chronology attempt, but the trainable adapter did not beat updated retrieval on exact API-fact memory. Retrieval and replay worked. The synthetic stale-doc positive control worked and cleared after a documentation fix. The synthetic patch-generation control grades candidate diffs by unit tests and only the structured patch passed. E5f generated a narrow MarkupSafe proxy-object patch from exposed source, passed an executable regression, and matched the historical future diff outcome while the no-op failed. E5g broadened public replay to two historical fixes: MarkupSafe `54bb00b` and Kilo `8e9a9bb`; both baselines failed, both historical patches passed, and both deterministic generated candidates passed their task regressions. The external update controller demonstrated versioning, rejection of unsafe/tampered candidates, and rollback in a reduced prototype. Structured symbol memory beat text retrieval on markupsafe symbol QA, but public stale-doc scans found no stale-doc issue to score.
- A1 external memory: exploration-positive in a synthetic repository-convention task, measurement-positive on public symbol/signature QA, positive on constrained code-generation proxies, positive on a constrained documentation-generation proxy, positive on exact public API-doc coverage lookup, and positive on a synthetic API-doc drift control. In E6a, gated structured external memory reached answer accuracy 1.000, denied the restricted public query, answered the authorized security query, and produced zero poisoned answers; ungated updated text retrieval reached answer accuracy 0.600 and produced one poisoned answer. In E6b, structured symbol memory answered 70/70 public symbol-file questions correctly across local `kilo` and `markupsafe` clones, while text file retrieval answered 12/70 correctly. In E6c, structured signature memory answered 23/23 defining signature-line questions correctly, while text signature lookup answered 22/23 correctly but used far more source-line storage. In E6d, structured signature memory generated 64/64 canonical Python call stubs correctly on `markupsafe`, while a name-only call-stub baseline generated 26/64 correctly. In E6e, structured signature memory generated 64/64 minimal Python function skeletons correctly on `markupsafe`, while a name-only skeleton baseline generated 0/64 correctly. In E6f, structured docstring memory generated 10/10 minimal documented Python function skeletons correctly on `markupsafe`, while a signature-only skeleton baseline generated 0/10 correctly. In E6g, structured docstring memory generated 10/10 Markdown API reference entries with source file, signature, and first docstring line correctly, while a signature-only reference baseline generated 0/10 correctly. In E6h, structured documentation-directive memory answered 4/4 API-doc coverage questions correctly for `docs/escaping.rst`, while source-symbol-only memory answered 0/4 correctly. In E6i, structured source/documentation consistency memory detected the stale missing-source directive 1/1, while doc-directive-only memory answered 0/1 correctly. These are useful alternative-family signals, not model-level code-generation, documentation-quality, semantic documentation drift, or architecture Pareto claims.

## Novelty-Risk Assessment

Most individual mechanisms are established: MoE/routing, quantization, low-rank adaptation, retrieval memory, replay, codebook compression, signing/rollback controls, and canary/security tests. Potential novelty would have to come from a rigorously measured local combination with explicit security and rollback gates. Current component evidence does not justify a novelty claim. The highest risk is terminology-driven novelty: renaming existing retrieval, adapter, or quantization ideas as a new “multi-dimensional weight” architecture without a measured Pareto gain.

## Expected Scaling Behavior

- Routing: dispatch overhead and load balance become more important as component banks grow; multi-label routing increases active components and traffic unless cache or batch coherence offsets it.
- Compression: metadata/codebook overhead dominates small tensors; uniform/groupwise quantization remains a strong baseline until task-level quality proves otherwise.
- Selective precision: row-level output-head and toy internal-MLP results may not transfer to real attention/MLP internals; real models need calibration, held-out tasks, and kernel-aware layouts.
- Lookup: exact indexing and caching may help large banks, but GPU transfer, occupancy, and batching will likely dominate real latency.
- Continual learning: retrieval and structured memory scale operationally before weight updates; adapter updates need replay, validation, lineage, signing, and rollback, which add latency and storage.
- External memory: structured repository memory can be safer and more exact than raw text retrieval for convention, symbol-file, signature-line, constrained call-stub, deterministic function-skeleton, first-line documented-skeleton, templated API-reference, exact API-doc coverage, and synthetic API-doc drift tasks when symbols, aliases, trust labels, scopes, documentation directives, source indexes, or consistency checks are explicit, but real repositories need extraction, update, conflict resolution, richer QA, code-generation integration, public documentation quality/drift evaluation, and security policy integration.
- Security: normalized pattern-based canary/injection tests and hand-authored semantic query heuristics do not scale to production adversaries; production requires identity integration, sandboxing, key management, immutable logs, broad semantic red-team extraction testing, and side-channel analysis.

## Confounders And Limitations

Most current experiments are synthetic. The DS4 audit is primary-source implementation evidence for activation-aware routed-expert quantization and streaming, but it is not a reproduced experiment in this repository. E3c/E3d/E3e use transformer-style sequence models, but the trained runs train only the output head and mostly quantize output-head rows. E3f quantizes an internal MLP matrix whose values remain deterministic toy initialization. E3g trains that internal MLP output matrix before quantization, but it is still a tiny ridge-trained toy layer. E3h uses a real pinned tiny-gpt2 checkpoint tensor, but measures matrix reconstruction error only and uses an extremely skinny embedding matrix where row metadata dominates. E3i adds a real-checkpoint local forward pass, but it uses deterministic token-id sequences. E3j uses natural-language text through pinned GPT-2 BPE files, but the model is still too tiny for useful accuracy evidence and the quantized tensor is still the same skinny embedding matrix. E3k uses an internal tensor with wider rows, but the tensor has only 16 parameters and no measurable KL sensitivity in this tiny checkpoint. E5c trains a toy low-rank classifier adapter, not a language-model LoRA. E6a uses a synthetic convention corpus and a frozen parametric proxy. E6b/E6c use real public source but only ask symbol-definition file and signature-line questions. E6d generates canonical call stubs from extracted signatures, E6e generates deterministic function skeletons from extracted signatures, E6f preserves first docstring lines in deterministic skeletons, E6g generates templated API reference entries, E6h maps exact Sphinx documentation directives to documentation files, and E6i detects one synthetic missing-source directive after a source rename, but none synthesizes executable arguments, executable behavior, semantic program logic, semantic documentation explanations, documentation quality judgments, or public semantic prose-drift judgments. E4c executes through ROCm-backed PyTorch, and E4d records ROCm transfer scaling, but both remain small Torch benchmarks rather than fused-kernel, power, occupancy, or useful-scale model results. The public-repository experiments evaluate changed-file retrieval, symbol-definition QA, stale-doc scans, and generated patch candidates against historical patch controls. E5g improves public patch breadth to two tasks, but the generated fixes are still narrow hand-written heuristics; the Kilo task is an executable source-level regression rather than a runtime leak detector; and this is not broad autonomous patch generation or a full upstream test-suite repair benchmark. The stale-doc public samples had zero positive issues. The security/update runs are smoke tests without real identity integration, sandboxing, production signing, broad red-team coverage, learned or model-assisted semantic extraction testing, sandbox escapes, or side-channel analysis. S2b improves obfuscated-pattern coverage but remains deterministic string normalization. S2c adds trust-label quarantine and a deterministic English query heuristic, so it protects only the covered alias and query classes. There is no real LoRA training, no ROCm kernel specialization, no power measurement, and no end-to-end code-generation evaluation.

## Recommendation

Overall recommendation: **revise**. Continue only with the cheapest experiments that address the failed assumptions:

Direction recommendations:

- H1 routing: **revise**. Keep contextual/multi-label routing as an engineering requirement, but do not claim hierarchy is efficient.
- H2 compositional storage: **stop for now** unless a real model matrix or task-quality setting shows uniform/groupwise quantization is insufficient.
- H3 selective precision: **revise**. Continue only with larger local real-model task-loss tests or tensors with measurable task sensitivity; tiny-gpt2 embedding and internal-tensor smoke tests do not show a Pareto win.
- H4 indexed lookup: **revise**. Continue only with larger ROCm benchmarks or lower-level ROCm kernels that can measure real transfer, occupancy, batching, and fusion.
- H5 continual evolution: **revise**. Prefer retrieval/structured memory first; allow adapter work only behind update-controller gates and after retrieval fails.
- A1 external memory: **expand**. Move from exact API-doc coverage, templated API-reference generation, and synthetic API-doc drift detection to richer public-repository QA/code-generation tasks with extracted symbols, docs, trust metadata, conflict resolution, public documentation quality/drift checks, and retrieval baselines.
- Security architecture: **revise and harden**. Current gates are useful smoke tests, including simple obfuscation coverage and a small semantic-query heuristic, but far from production proof.

Cheapest next experiment that would reduce the most uncertainty:

1. Run H3 on a larger local open model or a tensor with measurable held-out task sensitivity, preferably still with natural-language held-out task loss.
2. Expand E4c/E4d into a larger ROCm benchmark that records occupancy, transfer behavior, kernel fusion, decompression, and model-layer dispatch.
3. Expand E5g beyond hand-authored heuristics toward upstream test-suite repair tasks and less pattern-specific patch generation.
4. Expand E6i from synthetic API-doc drift detection into public repository-memory QA/code-generation and documentation quality/drift tasks before attempting model integration.
5. Prototype activation-residual caches or ephemeral per-repository adapters only after retrieval-first baselines fail on a measurable task.
