## Important Results

- E1: hierarchy preserved synthetic route accuracy and used fewer toy active components than flat contextual routing, but had higher p95 routing latency.
- E1b: single-label contextual routing had mixed-route recall 0.500. Naive multi-label routing reached route recall 1.000 but added 0.258 false positive routes/sample and adversarial exact accuracy was 0.735. Latent-centroid multi-label routing reached exact set accuracy 1.000 but p95 latency rose to 2.779 ms.
- E2: uniform int8 dominated tested compositional methods on the synthetic matrix, including explicit Kronecker rank-1 and Tensor Train 4D baselines.
- E2 product quantization: product-quantized rows used 4288 bytes with MSE 30.298383 versus uniform int8 at 4108 bytes with MSE 0.004992, so the codebook baseline was rejected after overhead accounting.
- E2 tensor baselines: Kronecker rank-1 used 544 bytes but had MSE 59.978289; Tensor Train 4D used 4704 bytes with MSE 30.645373, losing to uniform int8 on both bytes and error.
- E3: causal ranking found planted carriers, but causal FP32 protection did not beat random protection on output MSE.
- E3b: groupwise int4 reduced MSE versus global int4, and output-error-selected FP32 protection beat random protected controls at the same 144-byte budget on a synthetic validation set.
- E3c: in a random tiny transformer-style LM with held-out prompts, output-error-selected FP32 output-head rows reduced held-out logit MSE to 0.007662 at 1368 bytes versus random mean 0.015225 and groupwise int4 0.018951.
- E3d: in a ridge-trained tiny transformer-style LM, trained held-out NLL improved from 4.417 to 1.920 versus untrained. Output-error-selected FP32 rows had held-out logit MSE 0.003384 at 1368 bytes versus random mean 0.004833, but the best random NLL draw was slightly better than selected.
- E3e: selected BF16-like carrier rows had held-out logit MSE 0.003354 at 1080 bytes versus random BF16 mean 0.005185. Selected FP32 carrier rows had 0.003353 at 1368 bytes, so FP32 gave negligible extra logit-MSE benefit over BF16-like carriers in this toy output-head run.
- E3f: internal MLP output rows selected by train-prompt output error had held-out logit MSE 0.001214 at 1952 bytes versus random FP32 protected mean 0.001845 and groupwise int4 0.002137. NLL remained mixed: selected NLL was 1.915895 while groupwise int4 and random protected controls were lower in this seed.
- E3g: after training the internal `w_mlp_out` matrix itself, selected BF16-like protected rows had held-out logit MSE 0.450158 at 1568 bytes versus random BF16 mean 0.644747 and groupwise int4 0.767562. Selected FP32 rows had similar logit MSE 0.450777 at 1952 bytes, so FP32 did not improve the logit-MSE tradeoff over BF16-like rows in this seed.
- E3h: on pinned public `sshleifer/tiny-gpt2` tensor `transformer.wte.weight` with shape `[50257, 2]`, selected BF16/FP32 protected rows beat matched random controls on reconstruction MSE, but row-wise groupwise metadata dominated storage. Groupwise int4 used 653341 bytes and selected BF16 used 654365 bytes versus full FP32 at 402056 bytes; uniform int4 used 50269 bytes but had higher reconstruction MSE. This is real-checkpoint evidence against a naive protected-row layout for skinny matrices.
- E3i: on the same pinned tiny-gpt2 tensor with a local minimal GPT-2 forward pass, selected BF16/FP32 protected rows did not beat matched random protected-row means on held-out KL. Selected BF16 KL was 0.000000885 versus random BF16 mean 0.000000842; selected FP32 KL was 0.000000860 versus random FP32 mean 0.000000839. This is real-checkpoint task-loss evidence against the current calibration-row selector for this tensor/workload.
- E3j: on byte-level GPT-2 BPE tokenized natural-language repository/code prose, selected BF16 protected rows had held-out KL 0.000000178 versus random BF16 mean 0.000000580, and selected FP32 protected rows had held-out KL 0.000000214 versus random FP32 mean 0.000000581. They did not beat the best random draws and still used 653597/653725 bytes versus full FP32 at 402056 bytes and uniform int4 at 50269 bytes, so this improves task realism but still does not show a storage Pareto win.
- E3k: on internal tensor `transformer.h.0.mlp.c_fc.weight` with shape `[2, 8]`, all precision policies had held-out KL 0.000000000 after numerical roundoff clamping. Uniform int4 used 20 bytes, groupwise int4 used 32 bytes, selected BF16 used 52 bytes, selected FP32 used 68 bytes, and full FP32 used 64 bytes. This fixes the skinny-embedding storage confound but finds no task-loss signal for selective precision on the tiny internal tensor.
- E4: dense/exact lookup remained strongest at useful recall for tested bank sizes.
- E4b: with full-path simulation, exact indexing beat dense routing at larger banks and cached exact routing improved repeated-query end-to-end latency; hash indexing saved memory but had poor recall and worse latency.
- E4c: Torch batched exact routing requested `device=rocm` and the current project PyTorch runtime reports `torch 2.12.1+rocm7.2`, HIP `7.2.53211`, `rocm_available=1`, `accelerator_backend=rocm`, and `uses_rocm_transfer=1`. PyTorch exposes the ROCm execution device as type `cuda`. At bank 512, p50 end-to-end was 0.5172 ms with recall 1.000 and p95 had a 30.1241 ms outlier; at bank 2048, p50 was 0.3888 ms and p95 was 0.6132 ms. This proves the ROCm path runs, but it is still a small benchmark rather than a GPU speed claim.
- E4d: ROCm transfer scaling recorded host-to-device, simple device dispatch, and device-to-host timing for 1, 8, and 32 MiB payloads. Best p50 host-to-device bandwidth was 22.917 GB/s at 8 MiB; best p50 device-to-host bandwidth was 28.715 GB/s at 8 MiB. This is a useful ROCm transfer baseline, but it does not measure occupancy, fusion, power, decompression kernels, or useful model layers.

## Blockers

The immediate blocker is stronger external functional evaluation and real packed-kernel quantization. T11c isolates much of the earlier T11 gain to width/allocation rather than adapters, but no T11 run dominates all measured quality/function/runtime/resource dimensions. D4 also lacks a paired documentation/source consistency benchmark, so that dimension remains unmeasured rather than claimed. Quantization has only been evaluated by reconstructing into FP32 PyTorch tensors; no packed int8/int4/BF16 runtime kernels were benchmarked.

## Decisions

- Start with small falsifying experiments.
- Defer integrated architecture.
- Treat DS4 as a selective-precision and routed-expert execution source case study, not evidence of semantic carrier pathways or a reproduced GPU Pareto result.
- The next phase must prioritize split-correct, optimizer-controlled architecture selection; the current snapshot is a reduced-research baseline plus real-training bootstrap/control/held-out-evaluation evidence, not a completed architecture result.
- Hugging Face corpus selection is exploratory-research-only. Mixed or unclear license metadata is recorded and does not block exploratory training, but the corpus is not approved for production, redistribution, or company deployment.

## Next Highest-Value Action

Train a matched byte-tokenizer/BPE model pair on D5 using the new fast `tokenizers` wrapper rather than the slow pure-Python BPE probe. Add or join timestamp-bearing source metadata before claiming temporal holdout. Run a stronger executable JavaScript benchmark and obtain or build a real paired documentation/source consistency corpus before claiming coding utility beyond D4 source-local probes. For compression, the next useful step is a packed-kernel int8/BF16 runtime measurement; the naive int4 protected-carrier path is not competitive with uniform int8 under current evidence.
