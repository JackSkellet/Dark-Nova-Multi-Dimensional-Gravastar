# Integrated Results

Status: historical reduced-experiment snapshot; superseded by later T11 split-correct training evidence.

No integrated architecture has been built because the component evidence gate has not been satisfied. E2, E3, and E4 did not support the speculative additions in their current form. Combining them now would make results harder to interpret.

At this reduced-experiment snapshot, the limitation-aware assessment kept the integration gate closed across a 44-record manifest. Later T11 split-correct training supersedes that snapshot: the current `results/research_assessment.json` reports a measured T11b frontier expansion, while still recording `dominates_dense_baseline=false` because T11b loses training throughput to T11a. E4c proves the ROCm-backed Torch dispatch path runs locally, E4d records ROCm transfer scaling, E5g is a limited positive for narrow public patch replay, S2c is a security smoke pass, E6a is an exploration-positive structured external-memory result on a synthetic repository-convention task, E6b is a real-public-repo measurement for symbol-file QA, E6c is a real-public-repo measurement for signature-line QA, E6d is a constrained public-repo call-stub generation proxy, E6e is a constrained public-repo function-skeleton generation proxy, E6f is a constrained public-repo documented-skeleton generation proxy, E6g is a constrained public-repo API-reference generation proxy, E6h is a real-public-repo API-doc coverage measurement, and E6i is a synthetic API-doc drift positive control. None of those reduced results alone was enough to justify an integrated architecture claim. Next integration candidates should be tested only after a larger task-sensitive H3 run, a useful-scale ROCm E4 benchmark, an upstream-test-suite E5 repair benchmark, or a richer real repository QA/code-generation memory benchmark produces a real advantage over retrieval, uniform/groupwise quantization, exact lookup, and other simple baselines.
